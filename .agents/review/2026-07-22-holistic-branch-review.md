# Holistic Review — `llm-led-generator` branch

**Scope:** the entire branch written by another agent — 141 commits, 86 files, ~35,061 insertions over `main` (`98abb13`), reviewed at head `89d194d`.

**Method:** multi-agent workflow (117 agents): 2 grounding agents extracted the approved goals/invariants from the plans and decision records; 12 dimension reviewers (security/auth, credentials, runtime/subprocess, durability, concurrency, web UI, rendering/mapping, packaging/CI, test quality, product-goal fit, docs drift, architecture) reviewed the head code goal-first; an independent agent re-ran the full recorded verification entry point; a completeness critic spawned 6 targeted gap finders; every non-nit finding then faced adversarial verification (3 independent refuter lenses for critical/major — reproduce, reachability, existing-guards — majority to survive; 1 for minor).

**Outcome: 87 unique findings → 61 confirmed (1 critical, 12 major, 47 minor, 1 downgraded to nit), 4 refuted, 22 polish nits.** The automated verification gate is fully green at head.

**Verdict in one paragraph:** This is unusually disciplined work for its size — the server-authoritative AI gating, credential migration, asset-banking atomicity, pinned-source packaging, and the Ollama-first slice's transport hardening all genuinely hold their stated invariants, and the recorded state in `.agents/` matches the code almost claim-for-claim. The confirmed defects cluster in four seams the process could not see: (1) the web UI layer is tested only by regex-on-source, so two head-breaking UI defects (the generation Review/Apply stage and the manual Target control) ship behind a green suite; (2) the flagship generation flow depends on launch-time CLI config that normal GUI launches never populate, which the CLI-launched acceptance runs masked; (3) startup reconciliation lacks per-job error containment, so one damaged job directory prevents app launch; (4) the Windows/Linux CI legs have never executed and contain errors that will fail their first run.

### MAJOR

**2. `am_configurator/ollama_client.py:156` — Cancelling an Ollama generation cannot interrupt the in-flight chat request, so the single admission slot stays held up to the full 180s deadline**

OllamaClient._request checks cancelled() only before opening the connection and after the blocking read completes; the urllib open/read itself blocks with timeout min(CHAT_TIMEOUT_SECONDS=180, remaining) and non-streaming /api/chat sends nothing until inference finishes. OllamaRecipeProvider.generate_attempt (recipe_provider.py:697) calls client.chat synchronously on the coordinator worker thread, which owns the process-wide OperationGate. The GGUF path solved exactly this with _loopback_exchange (recipe_provider.py:360-395): a request thread polled every 50ms plus an abort hook. The new primary local path has no equivalent, so cancel (procedural_generation.py:609) sets the event but the worker stays blocked in the socket read and the gate stays held; the same applies to an uncancellable /api/ai/test setup test (server.py:1882).

*Failure scenario:* User starts a local Ollama generation on a slow model, clicks Cancel after 5s; POST /api/lighting/jobs/<id>/cancel returns OK, but the worker is blocked inside response.read() until Ollama finishes or the 180s deadline expires. Every new generation or setup test during that window fails with 'another generation operation is already active' (409), so the user who cancelled to try a different prompt is locked out of AI for up to ~3 minutes.

### MINOR

**14. `am_configurator/server.py:1562` — Non-ASCII X-AM-Token header raises uncaught TypeError before the route try block**

_authorized() calls secrets.compare_digest(self.headers.get("X-AM-Token", ""), self.state.token). http.client decodes header bytes as latin-1, so a header byte >= 0x80 yields a non-ASCII str, and compare_digest raises TypeError for non-ASCII string operands. Both call sites (do_GET line 1659, do_POST line 1730) sit outside the route try/except, so the exception escapes the handler: socketserver prints a traceback to the console and drops the connection with no HTTP response instead of returning 403. It fails closed (no auth bypass), but the boundary that is supposed to answer every request cleanly instead crashes the handler thread on attacker-shaped input.

*Failure scenario:* Any local process sends GET /api/settings with header X-AM-Token: \xC3\xA9 (or any byte >= 0x80). compare_digest raises TypeError, the handler thread dies, a traceback is printed to the server console, and the client sees a dropped connection instead of the 403 every other bad-token request receives.

The auth and API boundary is in very good shape for this dimension. A single per-launch token gates every /api route in both do_GET and do_POST with a constant-time comparison, static serving is a fixed allowlist with no traversal surface, there are no CORS headers so the custom-header requirement doubles as CSRF protection, and CSP/nosniff/no-referrer are applied to every response. The newest Ollama slice holds the line: /api/ai/local/select is server-authoritative (strict one-field body, fresh fixed-loopback discovery supplies the digest, idle check, fingerprint invalidation on change), the hardened client refuses proxies/redirects and bounds and validates every response, and the auth requirement on all new routes is directly tested. Library asset serving is defense-in-depth done well: canonical UUIDs, resolve+relative_to containment, symlink/junction rejection, O_NOFOLLOW opens, hash/size integrity, and pathless public manifests; the private GGUF picker is genuinely unreachable from browser JS and its path never appears in a request or response. The browser side escapes all server-derived strings rendered via innerHTML and builds the model picker through DOM APIs. The findings that remain are edge robustness and consistency gaps, not breaches: a non-ASCII token header crashes the handler before the guarded try (fails closed), the non-AI/device error path returns raw exception text where the lighting/AI paths deliberately redact, and the legacy /api/settings/key//api/settings/test routes skip the strict-body/injected-store/idle-gate discipline their newer siblings follow. Residual accepted risk worth noting: the token travels once in the launch URL (browser argv/history exposure on multi-user machines, per the documented design) and the stdlib server has no read timeouts, leaving a local-only slowloris surface.

### credentials

The credential and settings layer is sound and notably disciplined for this dimension: schema v4 stays credential-free, v1/v2 plaintext keys migrate only through a verified exact read-back into the OS vault with rollback on every failure leg (vault-set failure, verify mismatch, settings-write OSError), the original v2 bytes are preserved fail-closed when migration cannot complete, and there is no plaintext fallback in resolve_xai_key. The new v3->v4 slice is correct and lossless per the approved goals — enabled/backend, API disclosure and fingerprint, library roots, and loop mode carry over, and a tested GGUF setup maps to the gguf advanced source with its fingerprint preserved while untested state resets to the ollama default; the v4 validator enforces the ollama/gguf pairing rules and every source or model change clears the local fingerprint. Credential hygiene is strong end to end: keys never appear in settings JSON, capability status, HTTP responses, or logs; backend exceptions are context-severed so a secret-bearing keyring error cannot chain; provider error text is redacted against the live key; the API fingerprint stores only a domain-separated double SHA-256, and files are written 0600 via mkstemp+atomic replace under an advisory lock. Test coverage in test_credentials.py and test_ai_routes.py exercises the important failure legs (unavailable vault, non-persisting vault, failed final write, no-echo of secrets). No critical or major defects were found. Residual risks are operational rather than exploitable: an unmigratable legacy file bricks all settings mutations behind one misleading error with no in-app repair, the migration writes an unvalidated v2 projection that a pathological ack string could later quarantine, and several legacy settings routes bypass the injected credential store — a latent test-isolation hazard that currently holds only by accident of fixture shape.

### runtime-subprocess

The local runtime and subprocess safety posture of this change is strong, and the new Ollama-first slice is the best-hardened part of it. The Ollama client is fixed to literal loopback constants with an empty ProxyHandler suppressing environment proxies, a redirect-refusing handler, a 1 MB response cap, strict 64-hex digest and model-id regexes, and layered cloud-alias exclusion (remote_model/remote_host metadata, :cloud suffix, completion-capability requirement); production code contains only GET /api/tags and POST /api/chat with zero model-management endpoints, verified by grep across am_configurator. Selection is genuinely server-authoritative — /api/ai/local/select accepts only model_id and persists the digest from a fresh discovery, generation admission re-verifies name+digest against live discovery, and the v4 schema clears the setup fingerprint atomically on any source/model change — and the recorded acceptance evidence (seven-model live inventory unchanged, one real recipe through ornith:latest) supports the /api/tags capabilities-field assumption that the eligibility filter depends on. The llama.cpp side retains solid attestation (pinned manifest hash, symlink-refusing O_NOFOLLOW hashing with before/during/after identity checks, exact-key attestation matching) and careful process handling (bounded 1 MB capture with overflow kill, terminate-to-kill escalation, idle shutdown, atexit); FFmpeg invocation is a fixed validated argv with -protocol_whitelist file, -nostdin, bounded ring-buffer diagnostics, and deadline/cancel polling. The four findings are all weak-guard minors, not invariant breaches on the main path: the Ollama chat read cannot be aborted on cancel and is only per-recv time-bounded (a hostile loopback squatter can wedge the single admission gate), llama-server can be orphaned on hard kill and its bearer token is argv-visible, and the xAI JSON transport lacks the redirect/proxy hardening its media and Ollama siblings have. Residual accepted risk worth naming: /api/chat is name-addressed, so a deliberate model swap in the seconds between digest verification and inference is undetectable (inherent to Ollama's API), and anything holding 127.0.0.1:11434 is trusted unauthenticated — both bounded by the fact that every model output still passes the local schema/quality validator and can never reach hardware without the manual Apply path.

### durability-jobs

The durability core of this branch is genuinely strong: asset banking uses an intent-file plus fsynced atomic-rename protocol whose orphan-adoption path I traced end to end; charged-operation costs are keyed by request_id/operation so repeated polling is idempotent and cancellation retains reported ticks; local retry bounds (2 retries after the initial attempt, deterministic sha256-derived per-attempt seeds) are enforced both in the coordinator loop and structurally in the manifest schema (attempt index capped at 0-2); startup recovery adopts fully banked procedural and video work only after per-asset hash verification and never replays paid API submissions (interrupted API jobs land in interrupted_api_no_replay, and video resume is poll-only). The v2-to-v4 settings and credential migrations verify vault read-back before rewriting and restore on failure, and the new Ollama provider maps its errors into the shared ProviderError codes so the coordinator's retry/cancel/failure semantics apply unchanged, with per-generation failures leaving readiness and selection intact. The weaknesses concentrate in reconciliation robustness and idempotence rather than the write path: library.reconcile() and the procedural reconcile_startup are all-or-nothing (one damaged job directory or read-only historical root aborts recovery and, because create_server calls reconcile_lighting unguarded, prevents the app from launching — empirically reproduced), and _recover_banked_animation plus the interrupted-status handling rewrite manifests they did not need to touch on every startup, which both drifts user-visible timestamps and breaks the v1 byte-preservation invariant for historical jobs. These are recovery-hardening gaps, not design flaws; fixing them is localized (per-job try/except mirroring the pattern the video coordinator already uses, a no-op/consistency check before rewriting, and gate acquisition in the procedural reconcile). Residual risk elsewhere in the dimension is low; the newest Ollama slice adds no new durability surface beyond the provider mapping, which is clean.

### concurrency

The admission architecture is fundamentally sound and matches the approved invariants: one process-wide OperationGate is genuinely shared by the setup test (server.py:1882), both generation coordinators, and legacy reconciliation; begin() is non-blocking, finish() is token-guarded and idempotent, cancel is ownership-checked at the gate, and the deferred-reconcile machinery (call_when_idle / wait_until_idle with a pending flag) self-corrects when it loses races. Per-job file-plus-thread locks in library.py make every manifest update atomic, readiness is recomputed server-side at each entry point, and fingerprint binding makes the select/settings TOCTOU windows (routes guarded only by an advisory is_active snapshot) fail closed rather than fail open. The ManagedLlamaServer idle-warm lifecycle uses a correct generation-counter timer pattern, and GGUF model attestation (O_NOFOLLOW, fstat identity before/after hashing) is careful against file TOCTOU. The most significant gap is in the newest Ollama-first slice: the hardened OllamaClient accepts a cancelled callback but cannot interrupt the blocking non-streaming chat, so cancelling the now-primary local backend leaves the single admission slot occupied for up to the 180s deadline — a responsiveness regression relative to the abortable llama.cpp exchange the same codebase already ships. The remaining findings are narrower: the procedural coordinator inverts cancel's stamp-then-check order, its reconciliation runs outside the gate and can transiently misclassify a live job, a rare post-launch failure path releases the gate under a running worker, and unsynchronized lazy singletons can duplicate the warm llama runtime. None of these threaten credentials, hardware safety, or paid-request replay; residual risk is concentrated in cancellation UX and rare race-window state mislabeling rather than data loss.

The web layer's architecture is sound and fits the product goals well: lighting_state.js is a clean pure-state module with real reducer semantics, routing lands on the manual workspace by default, the disabled first paint genuinely exposes no AI control (static hidden attributes plus aiReady() gating that fails closed when /api/ai/status is unreachable), and the restored Ollama Settings panel is one of the best-built parts — it uses safe DOM APIs instead of innerHTML, preserves selection across transient refresh failures, distinguishes service-down / no-models / model-removed states, and never surfaces paths, credentials, or raw service output. ### render-mapping

The deterministic rendering and mapping core is sound and closely fits the approved goals. The recipe contract is double-validated (strict provider-facing JSON schema plus an independent semantic validator shared by the xAI, llama.cpp, and Ollama providers), the renderer is exactly loop-periodic by construction (all layer kinds are integer-cycle periodic in phase, seeded PRNG for sparkle/noise, banker's-rounded 4x supersampling), and gif_durations distributes the 34 ms firmware step drift-free onto GIF centisecond ticks. The quality gate implements the approved plan's thresholds verbatim (non-zero motion, seam <= 1.25x adjacent + 0.01, per-density lit ratios, peak brightness > 180), device frame count/raster/duration are derived server-side from MODEL_FRAME_CAPS and _GIF_LAYOUTS via canonical_target_snapshot so the browser cannot override them, and mapping goes through the single shared frames_to_led_tracks core with a parity test; the qualification tool and dev CLI reuse the exact production validator, renderer, gate, and Ollama client, satisfying the no-divergent-transport goal. The material gaps found are: the banked "exact" raster/preview GIFs are 128-color quantized and therefore not exact for the CyberBoard 40x5 display target (mapped/applied LED data remains exact); the render/quality/encode phase runs with no deadline or cancellation while holding the process-wide operation gate, with progress frozen at 0/N and coordinator phases that don't match the UI's label map; and two small hygiene issues (vestigial loop_mode on procedural manifests, missing mapped-result guard the video path has). Residual risks worth noting but not filed as findings: worst-case schema-valid recipes make pure-Python rendering cost swing by two orders of magnitude, and LOCAL_OUTPUT_TOKENS=1536 may be tight for maximal 6-layer recipes on some tokenizers (a deterministic-retry-then-fail outcome, not a correctness break).
